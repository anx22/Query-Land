## Harte Einschätzung

Der richtige Nachbau ist **kein Semrush-Klon**, sondern ein **Google-first SEO Operating System** mit Semrush-/Ahrefs-/XOVI-/Similarweb-Logik.

Die Marktführer verkaufen vor allem drei Dinge: riesige Fremddaten-Indizes, gute Verdichtung, klare Workflows. Für deine eigenen Plattformen brauchst du zuerst **eigene Wahrheit**: Google Search Console, Analytics/Matomo, Serverlogs, eigener Crawl, PageSpeed/Lighthouse, SERP-Snapshots. Google selbst beschreibt SEO im Kern als Hilfe für Suchmaschinen, Inhalte zu verstehen und für Nutzer auffindbar zu machen; es gibt laut Google kein Geheimrezept, das automatisch Platz 1 erzeugt. Genau deshalb ist der „geheime Milliarden-Moat“ der Tools für interne Optimierung kleiner, als deren Marketing klingt.

Der echte Moat der Großen liegt in Datenmasse: Semrush spricht von 27,9 Milliarden Keywords, 808 Millionen Domains, 43 Billionen Backlinks, 500 TB Trafficdaten und Clickstream-/SERP-/Crawler-Quellen; Ahrefs nennt unter anderem 12 Milliarden Keywords, 17,9 Milliarden Content-Seiten, Backlink-Refreshes alle 15 Minuten und bis zu 8 Milliarden gecrawlte Seiten pro Tag; Similarweb beschreibt ein Hybridmodell aus First-Party-Analytics, Contributory Network, Partnerdaten, Public Data Extraction und ML-Modellierung. Das kopierst du sinnvoll als **Methodik**, aber baust die Datenwahrheit für deine Domains selbst.

# Hauptmodule des internen SEO-OS

## 1. Project Control & Data Governance

**Zweck:** Jede Domain wird als steuerbares SEO-Projekt behandelt: Domain, Subdomain, Verzeichnis, Land, Sprache, Gerät, Suchmaschine, Wettbewerber, Zielseiten, Business-Prioritäten, Tags, Verantwortliche.

**Was Marktführer machen:** Semrush, XOVI, Ahrefs und Similarweb arbeiten projekt-/kampagnenbasiert: Domain anlegen, Keywords wählen, Wettbewerber setzen, Crawls planen, Reports erzeugen, Fortschritt historisieren. Similarweb Rank Tracker verlangt zum Beispiel Website, Competitors, Search Engine, Device, Geo, Language und Keywords/Tags als Kampagnenlogik.

**Was dein Tool können muss:**

| Funktion       | High-Level-Logik                                                         |
| -------------- | ------------------------------------------------------------------------ |
| Project Scope  | Domain, Subdomain, Folder, URL-Set                                       |
| Market Scope   | Land, Sprache, Device, Suchmaschine                                      |
| Competitor Set | manuell + automatisch über SERP-Overlap                                  |
| Goal Model     | Traffic, Leads, Revenue, Sichtbarkeit, technische Qualität               |
| Source Trust   | Google-Daten hoch, Drittanbieter geschätzt, SERP-Snapshots beweisführend |
| Change Log     | Content-Änderungen, technische Releases, Crawl-Diffs, Ranking-Deltas     |

**Arbeitsmethode:**
Jedes Projekt bekommt ein **SEO Control Sheet**: Was ist wichtig? Welche Seiten bringen Business? Welche Keywords zählen wirklich? Welche Konkurrenz ist real? Genau hier beginnt der Vorteil gegenüber generischen Tools: Semrush kennt deine Business-Prioritäten nur oberflächlich, dein internes Tool macht sie zur Rechenlogik.

**Guardrail:**
Globale Sichtbarkeit ist nett. **Projektbezogene Sichtbarkeit ist entscheidend.** Baue also keinen Vanity-OVI, sondern einen eigenen **Project Visibility Score** aus deinem Keyword-Set, deinen Märkten und deinen Zielseiten.

## 2. Google Truth Layer

**Zweck:** Google-Daten werden zur belastbaren Wahrheitsebene für eigene Domains.

**Datenquellen:**

| Quelle                    | Nutzen                                                                 |
| ------------------------- | ---------------------------------------------------------------------- |
| Google Search Console API | Queries, Pages, Countries, Devices, Clicks, Impressions, CTR, Position |
| URL Inspection API        | Indexierungsstatus, Canonical-Hinweise, Google-Index-Version           |
| PageSpeed Insights API    | Lighthouse-Labdaten, CrUX-Felddaten, Performance-/SEO-Hinweise         |
| GA4 / Matomo              | Sessions, Conversions, Engagement, Revenue                             |
| Serverlogs                | Googlebot-Besuche, Crawl-Frequenz, Crawl-Waste, Bot-Verhalten          |

Die Search Console API kann nach Dimensionen wie Query, Page, Country, Device und Search Appearance gruppieren und liefert Clicks, Impressions, CTR und Position; die Ergebnisse sind nach Klicks sortiert und durch API-/Top-Row-Logik begrenzt. Daraus folgt: täglich speichern, historisieren, fehlende Longtail-Ränder akzeptieren.

Die URL Inspection API liefert Informationen zur von Google indexierten Version einer URL, etwa Indexed-/Indexable-Status, aber sie testet laut Google keine Live-URL in Echtzeit. Daher nutzt du sie als Index-Beweis, ergänzt durch eigenen Crawl und Live-Checks.

PageSpeed Insights kombiniert Lighthouse-Labdaten und echte Felddaten aus CrUX; Google weist zugleich auf Änderungen bei CrUX-Daten in der PSI API hin, also muss deine Architektur CrUX/PSI modular behandeln.

**Was dein Tool können muss:**

| Fähigkeit           | Umsetzung high-level                                     |
| ------------------- | -------------------------------------------------------- |
| Daily GSC Snapshots | Query/Page/Country/Device/Search-Type speichern          |
| URL Index Sampling  | wichtige URLs regelmäßig per URL Inspection prüfen       |
| Page Value Model    | Traffic + Conversion + Ranking-Potenzial + Business-Wert |
| Query-Page Matrix   | Welche Suchanfrage landet auf welcher URL?               |
| CTR Gap             | Position vs. erwartbare CTR vs. tatsächliche CTR         |
| Decay Detection     | Seiten, die langsam Impressionen/Klicks verlieren        |
| Release Overlay     | SEO-Veränderungen gegen Website-Deployments legen        |

**Innovation für dich:**
Baue ein **SEO Ledger**: Jede URL bekommt eine Zeitachse aus Crawlzustand, GSC-Leistung, Content-Änderung, Ranking, internen Links, Indexstatus und Conversionwert. Das ist mächtiger als ein Dashboard, weil du Ursache und Wirkung sehen kannst.

## 3. Crawl & Technical Site Audit

**Zweck:** Dein eigener Crawler wird das Rückgrat. Er erkennt technische SEO-Probleme, URL-Zustände, interne Verlinkung, Render-Probleme und strukturelle Schwächen.

**Was Marktführer machen:**
Semrush Site Audit ist ein Website-Crawler mit 140+ technischen und Onpage-Checks, JavaScript-Rendering, Crawl-Scopes für Domain/Subdomain/Subfolder und planbaren Recrawls.
Ahrefs Site Audit prüft 170+ technische und Onpage-Probleme; der Health Score basiert auf dem Anteil interner URLs ohne Errors.
XOVI Onpage analysiert unter anderem Title, Server, Snippets, Duplicate Content, externe Links, Bilder, Ankertexte, Architektur, Pagespeed und nutzt einen JavaScript-Crawler; der Advisor übersetzt Ergebnisse in priorisierte Aufgaben.

**Open-Source-Bausteine:**
Scrapy ist ein asynchrones Crawling-Framework mit Concurrency, Robots.txt-Unterstützung, Crawl-Politeness, Depth-Limits, Sitemap-Unterstützung, Pipelines und Exporten.
Crawlee bietet JavaScript/Python-Crawling, Browser-Automation, Proxy-/Blocking-Handling und PlaywrightCrawler für gerenderte Seiten.
SiteOne Crawler ist ein Open-Source-Rust-Crawler mit SEO-, Security-, Accessibility-, Performance-, CI/CD- und AI-Markdown-Export-Fokus.
Lighthouse CI automatisiert Lighthouse-Prüfungen pro Commit und eignet sich als Qualitätsgate gegen Performance-/SEO-Regressions.

**Was dein Tool können muss:**

| Bereich         | Checks                                                             |
| --------------- | ------------------------------------------------------------------ |
| Discovery       | Start-URL, interne Links, Sitemaps, GSC-URLs, Logs                 |
| Fetch           | Statuscodes, Redirect Chains, Timeouts, MIME Types                 |
| Indexability    | robots.txt, meta robots, x-robots, canonical, noindex              |
| Rendering       | HTML vs. JS-rendered DOM, fehlende Inhalte, broken JS              |
| Metadata        | Title, Description, H1/H2, OG, Twitter, Canonical                  |
| Structure       | interne Links, Tiefe, Orphans, Hubs, Pagination                    |
| Duplicates      | identische Titles, Meta, Content-Fingerprints                      |
| International   | hreflang, canonical/hreflang-Konflikte                             |
| Media           | Alt-Texte, Bildgrößen, lazy-loading                                |
| Structured Data | Schema.org, JSON-LD, Validität, Entity-Signale                     |
| Performance     | Lighthouse, Core-Web-Vitals-Nähe, TTFB, LCP, CLS, INP              |
| Security/Trust  | HTTPS, Mixed Content, HSTS, alte Assets                            |
| AI Readiness    | klare Fakten, Autoren, Quellen, semantische Struktur, Crawlbarkeit |

**Arbeitsmethode:**
Jeder Crawl erzeugt drei Ebenen:

1. **URL Inventory**: Welche URLs existieren?
2. **Issue Instances**: Welche Probleme existieren wo?
3. **Crawl Diff**: Was hat sich seit letztem Crawl verändert?

**Bester Score-Ansatz:**
Ahrefs’ Health-Score-Idee ist als Basis gut, aber für dich zu grob. Besser:

```text
Technical Priority =
Issue Severity
× affected URL business value
× index/ranking relevance
× fix confidence
× repetition pattern
```

**Guardrail:**
Ein Fehler auf einer unwichtigen Archivseite ist weniger wert als ein mittelgroßer Fehler auf einer Money Page. Dein Audit muss **Business-Kontext** verstehen.

## 4. Indexability & Crawl Budget Observatory

**Zweck:** Dieses Modul beantwortet: Welche URLs kann Google finden, crawlen, rendern, canonicalisieren, indexieren und ranken?

Google beschreibt selbst, dass Google Search automatisch crawlt, Seiten vor allem über Links findet, Sitemaps helfen und Google dieselben Ressourcen wie ein Browser braucht, inklusive CSS/JS/Bildern.

**Was dein Tool können muss:**

| URL-State     | Bedeutung                         |
| ------------- | --------------------------------- |
| Discovered    | URL wurde gefunden                |
| Crawlable     | URL kann gecrawlt werden          |
| Rendered      | HTML/JS liefert sinnvollen Inhalt |
| Canonicalized | Canonical-Ziel ist sauber         |
| Submitted     | URL steht in Sitemap / GSC        |
| Indexed       | Google kennt sie im Index         |
| Ranking       | URL rankt für Queries             |
| Performing    | URL erzeugt Klicks/Conversions    |

**Arbeitsmethode:**
Baue eine **URL Lifecycle State Machine**:

```text
Discovered → Crawlable → Rendered → Canonicalized → Submitted → Indexed → Ranking → Performing
```

Jede URL bekommt einen Status und einen Engpass. Beispiel:

| Engpass                        | Aktion                                              |
| ------------------------------ | --------------------------------------------------- |
| gefunden, aber blockiert       | robots/noindex/canonical prüfen                     |
| gecrawlt, aber nicht indexiert | Thin Content, Duplicate, Canonical, Qualität prüfen |
| indexiert, aber keine Rankings | Keyword-/Intent-Fit prüfen                          |
| Rankings, aber schlechte CTR   | Title/Snippet/SERP-Feature prüfen                   |
| Klicks, aber keine Conversion  | Content-/UX-/Intent-Problem prüfen                  |

**Innovation:**
Markttools zeigen oft nur Listen. Dein System soll eine **Pipeline-Diagnose** liefern: An welcher Stufe verliert eine URL SEO-Wert?

## 5. Search Performance Intelligence

**Zweck:** GSC-Daten werden in strategische Handlungslogik übersetzt.

Similarweb SEO Overview bündelt organische Suchdaten, Keywords, Seiten, Intents, Wettbewerber, Ranking-Verteilungen und Gap-Analysen in einer Performance-Sicht.
Similarweb Website Keywords zeigt Keywords für Domains, URLs, Subdomains und Prefixes, inklusive Traffic-Anteil, Wettbewerbsvergleich, Branded/Non-Branded, Organic/Paid, Device, Zeitraum und Gewinner-/Verlierer-Logik.

**Was dein Tool können muss:**

| Funktion               | Zweck                                         |
| ---------------------- | --------------------------------------------- |
| Query-Page Matrix      | Welche Query erzeugt welche URL-Leistung?     |
| Page Performance Map   | Welche URLs gewinnen/verlieren?               |
| CTR Opportunity        | Position gut, CTR schwach                     |
| Impression Opportunity | viele Impressions, Position schwach           |
| Decay Detection        | schleichender Sichtbarkeitsverlust            |
| Query Drift            | Seite rankt plötzlich für anderes Intent-Feld |
| Brand vs. Non-Brand    | echte SEO-Leistung statt Markenrauschen       |
| Device Split           | mobile/desktop Unterschiede                   |
| Country Split          | Marktunterschiede                             |
| Search Appearance      | Rich Results, Videos, Product Snippets etc.   |

**Arbeitsmethode:**
Nicht einzelne Keywords optimieren, sondern **Page-Query-Cluster**. Eine URL ist selten „für ein Keyword“ relevant, sondern für ein Suchfeld.

**Beste interne KPI:**

```text
Page Opportunity =
(Impression Potential × CTR Upside × Position Upside)
× Business Value
× Fix Confidence
× Freshness Need
```

**Guardrail:**
Für eigene Domains ersetzt GSC viele teure Fremdmetriken. Fremdtools schätzen, GSC misst. Nutze Schätzungen für Konkurrenz, nutze Google-Daten für eigene Entscheidungen.

## 6. Keyword Universe & Topic Graph

**Zweck:** Ein kuratiertes Keyword-/Themenuniversum für deine Plattformen, statt sinnloser Milliarden-Datenbank.

Semrush Keyword Magic arbeitet mit 27,3+ Milliarden Keywords, gruppiert verwandte Begriffe thematisch, zeigt Volumen, Difficulty, SERP Features, Fragen und Filter.
Ahrefs Keywords Explorer, Site Explorer und Content Gap liefern Keyword-Ideen, Wettbewerbsüberschneidungen und Lücken; Ahrefs Content Gap vergleicht bis zu 10 Wettbewerber und zeigt Keywords, für die diese ranken und die Zielseite/Domain nicht.

**Was dein Tool können muss:**

| Quelle              | Rolle                                      |
| ------------------- | ------------------------------------------ |

**Datenmodell high-level:**

```text
Keyword
→ Intent
→ Topic Cluster
→ Entity Cluster
→ Funnel Stage
→ Target URL
→ Competitors Ranking
→ SERP Features
→ Opportunity Score
```

**Keyword-Difficulty intern:**
Baue eine eigene Difficulty, aber transparent:

```text
Internal Difficulty =
SERP Strength
+ competitor authority
+ backlink depth of ranking URLs
+ content depth
+ SERP feature compression
- own topical authority
- own page relevance
```

**Innovation:**
Das Keyword-Modul wird kein Keyword-Tool, sondern ein **Topic Graph**. Keywords sind nur Beobachtungspunkte. Die eigentliche Einheit ist: Thema, Intent, URL, Entität, SERP-Typ.

**Zweck:** Du brauchst eigene, beweisbare SERP-Beobachtung für dein priorisiertes Keyword-Set.

Semrush Position Tracking überwacht tägliche Rankings für bestimmte Keywords nach Device, Ort und Suchmaschine und berücksichtigt inzwischen auch Google AI Mode, ChatGPT Search und AI Overviews.
Similarweb Rank Tracker trackt tägliche Ranking-Veränderungen, Wettbewerber, SERP Features, Kannibalisierung und Content-Ideen; Kampagnen können nach Domain/Subdomain/Folder, Device, Geo, Sprache und Suchmaschine angelegt werden.
Ahrefs Rank Tracker überwacht Google Rankings auf Desktop und Mobile über 190 Standorte.

**Was dein Tool können muss:**

| Funktion               | Logik                                                   |
| ---------------------- | ------------------------------------------------------- |
| Keyword Set Tracking   | daily/weekly nach Priorität                             |
| Geo/Device Split       | Land, Stadt, Sprache, Mobile/Desktop                    |
| SERP Snapshot          | Top 10/20/50 speichern                                  |
| SERP Feature Detection | AI Overview, Featured Snippet, Video, Image, Local, PAA |
| Own URL Mapping        | welche eigene URL rankt?                                |
| Cannibalization        | mehrere eigene URLs im gleichen SERP-Feld               |

| Volatility             | Ranking-Drops, neue Wettbewerber, Intent-Wechsel        |

**Arbeitsmethode:**
Für jedes Keyword speicherst du nicht nur Position, sondern den **SERP-Fingerprint**:

```text
SERP Fingerprint =
Top URLs
+ Domains
+ Page Types
+ SERP Features
+ AI/Answer Elements
+ local/video/image/product modules
+ intent class
```

**Innovation:**
Das Tool erkennt, wann ein Keyword strategisch kippt. Beispiel: Früher rankten Blogartikel, jetzt Produktseiten. Dann ist Content-Optimierung falsch, und eine andere Seitenart muss entstehen.

**Guardrail:**
SERP-Daten sauber beziehen: offizielle APIs, lizenzierte SERP-Provider, oder kontrollierte interne Abrufe mit klarer Rate-Limit- und Legal-Policy. Fragiles Massenscraping ist ein Zeitgrab.

## 8. Content Opportunity & Onpage Intelligence

**Zweck:** Aus Messung wird Arbeit. Dieses Modul entscheidet, welche URL warum zuerst angefasst wird.

Semrush On-Page-Logik arbeitet mit Landing Pages und Keywords und erzeugt Handlungsideen; XOVI koppelt Onpage-Analyse an den Advisor und Textoptimierung gegen Top-10-Ergebnisse, semantische Begriffe, Häufigkeit und Textlänge.
Ahrefs Opportunities/Content-Gap-Logik deckt Content-, Link- und technische Chancen auf; Content Gap zeigt explizit Keywords, für die Wettbewerber ranken und du nicht.

**Was dein Tool können muss:**

| Opportunity-Typ     | Erkennung                                        |
| ------------------- | ------------------------------------------------ |
| Low Hanging Fruit   | Position 4–20 + gute Impressions + passende URL  |
| CTR Fix             | gute Position + schwache CTR                     |
| Content Refresh     | historische Gewinnerseite verliert langsam       |
| Cannibalization     | mehrere eigene URLs kämpfen um dieselbe Query    |
| Missing Intent      | Seite rankt, aber Suchintention passt nur halb   |
| Content Gap         | Wettbewerber decken Themen ab, du nicht          |
| Entity Gap          | wichtige Entitäten/Begriffe fehlen               |
| Internal Link Boost | wichtige Seite hat zu wenig interne Autorität    |
| Snippet Opportunity | Title/Description passen nicht zur SERP-Sprache  |
| Pruning Candidate   | URL kostet Crawl/Indexbudget, bringt keinen Wert |

**Arbeitsmethode:**
Jede Empfehlung bekommt:

```text
Evidence
→ Impact
→ Effort
→ Confidence
→ Affected URLs
→ Suggested Change
→ Validation Metric
```

**Beispiel einer guten Empfehlung:**

```text
URL /seo-tool-vergleich/ verliert seit 21 Tagen Impressions im Cluster "semrush alternative".
Ranking fällt von Ø 6,2 auf Ø 11,8.
SERP zeigt jetzt mehr Vergleichstabellen und Preisabschnitte.
Empfehlung: Vergleichssektion + Preis-/Feature-Tabelle + aktualisierte 2026-Signale ergänzen.
Validierung: Impressionen, Position, CTR für Cluster nach 14/28 Tagen.
```

**Guardrail:**
LLMs formulieren Briefings und Varianten. Die Entscheidung kommt aus GSC, Crawl, SERP und Businesswert. Sonst baust du ein hübsches Textspielzeug.

**Was dein Tool können muss:**

**Innovation:**

| Expiry         | Wann wird die Empfehlung neu geprüft?            |
| Status         | Open, planned, implemented, validated, dismissed |

**Prioritätsmodell:**

```text
Advisor Priority =
Impact
× Confidence
× Business Value
× Urgency
÷ Effort
```

**Beste Task-Klassen für V1:**

1. Indexierungsprobleme auf wichtigen Seiten
2. technische Fehler mit Ranking-/Crawl-Wirkung
3. CTR-Chancen
4. Low-Hanging-Fruit-Keywords
5. Kannibalisierung
6. interne Linkchancen
7. Content-Refresh-Kandidaten
8. Broken Backlink Targets
9. AI-Citation-Gaps
10. Reporting-Anomalien

**Guardrail:**
Jede Empfehlung braucht eine Validierung. Ohne Vorher/Nachher-Messung sammelst du Aufgaben wie Briefmarken.

# Open-Source-Stack als realistische Grundlage

| Bereich           | Kandidaten                               | Rolle                                         |
| ----------------- | ---------------------------------------- | --------------------------------------------- |
| Crawling          | Scrapy, Crawlee, SiteOne, optional Nutch | URL Discovery, HTML/JS Fetch, Crawl Pipelines |
| Browser Rendering | Playwright                               | JS-Seiten, DOM-Vergleich, Screenshots         |
| Performance       | Lighthouse, Lighthouse CI, PageSpeed API | Lab-/Regression-Checks                        |
| Storage           | Postgres + Object Storage                | System of Record + Crawl-Artefakte            |
| Analytics Scale   | ClickHouse später                        | SERP-/GSC-/Crawl-Historien                    |
| Search            | OpenSearch/Meilisearch später            | URL-/Issue-/Content-Suche                     |
| Graph             | Neo4j/Graph DB optional                  | URL-Linkgraph, Topic Graph, Authority Flow    |
| Queue             | Redis/BullMQ/Celery                      | Crawl-/Rank-/Report-Jobs                      |
| Web Corpus        | Common Crawl                             | grobe externe Content-/Link-Signale           |
| AI Layer          | LLM + RAG über eigene Evidence           | Briefings, Erklärungen, Tickets               |

Mein Take: **Scrapy/Crawlee + Playwright + Lighthouse + Postgres reichen für V1.** ClickHouse, Graph DB und Common Crawl erst dann, wenn reale Datenmenge oder Analysebedarf es erzwingen.

# Strategische Wahrheit

Du kannst für eigene Websites sehr viel nachbauen:

| Bereich                            |           Realistisch intern erreichbar |
| ---------------------------------- | --------------------------------------: |
| Technical Audit                    |                                  85–95% |
| GSC Search Intelligence            |                                    90%+ |

| Traffic Estimation fremder Domains |                                  20–50% |
